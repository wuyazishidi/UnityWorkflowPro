严格遵守 `specs/constitution.md` 与 `CLAUDE.md`。除非遇到下面"暂停点"，否则**自驱动跑完整条线，不要中途停下等我确认**。

## 流程
1. **澄清（仅在必要时）**：仅当需求存在会**实质改变实现**的歧义时，用 AskUserQuestion 提最多 2-3 个针对性问题；否则按合理默认推进，并把假设写进 spec。
2. **规约**：按 `/spec-new` 逻辑创建 `specs/NNN-*.md`（目标/范围/接口/约束/**可自动化验证的验收**）。
3. **计划 + 任务**：生成 `*.plan.md` 与 `*.tasks.md`（每个任务可独立验证；必须含"补 EditMode 测试"与"跑 DoD"两项）。
4. **实现**：按任务逐个实现。遵守架构约定——业务逻辑抽**纯类/可注入**（用 VContainer 注册、UniTask 处理异步），MonoBehaviour 仅作组合根；为可测逻辑写 EditMode 测试。
5. **门禁循环（关键）**：跑 `scripts/dod.ps1`。**红了就修，循环直到全绿**（编译 Success + 测试 result=PASS）。不要在未绿时声称完成。若 Unity 不可用导致门禁无法运行，明确标注"未验证"并停下告诉我。
6. **审核包**：全绿后，产出一段**给开发者在 Unity 里审核**的清单：
   - 改了哪些文件 / 新增了什么
   - **在 Unity 哪里看**：哪个场景、把哪个组件挂到哪个 GameObject、进入 Play 后按什么键/看什么现象
   - 测试结果（passed/failed）
   - 已知限制与建议的后续

## 暂停点（必须停下问我）
- 需求有实质歧义（见第 1 步）
- 需要引入新的第三方包/大改架构
- 门禁因环境（Unity 未开）无法运行

## 完成标准
DoD 全绿 + 勾选 spec 验收清单 + 给出审核包。最后一句话明确：DONE 还是 还有什么待办。
