# Specs — 规约驱动开发（SDD）

本目录是工程的“真理来源”。**先写 spec，再让 AI 实现**，用结构化的契约消除 AI 的幻觉空间，让产出可预期、可验证。

## 怎么用

1. 复制 `_TEMPLATE.md`，命名为 `NNN-功能名.md`（如 `001-player-movement.md`）。
2. 把功能写清楚：目标、范围、接口/数据结构、约束、验收标准。一份密集的 spec 胜过几千行让 AI 自己揣摩的代码。
3. 让 AI 按 spec 实现 → 编译 → 验证（见根目录 `CLAUDE.md` 第 2 节的强制流程）。
4. 实现完成后，回到 spec 勾掉验收清单；spec 与代码不一致时，以 spec 为准先对齐。

## 原则

- **一个 spec 一个功能**，范围窄、可独立测试。
- 明确写出：**数据类型、方法名、性能/平台约束** —— 不给 AI 留模糊地带。
- 验收标准要可机器/人工验证（最好能映射到一条编译/运行/断言命令）。

## 编号约定

- `001`–`099`：核心系统（角色、战斗、UI 框架…）
- `100`+：具体玩法 / 内容
- 已废弃的 spec 加前缀 `_archived-` 或移入 `specs/_archive/`
